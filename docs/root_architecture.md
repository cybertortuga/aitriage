# Архитектура и Pipeline AITriage (Under the Hood)

Здесь описано, как именно данные проходят через ядро `AITriage` при запуске сканирования. Инструмент спроектирован для работы `O(N)` и полностью автономен (не выполняет внешних API запросов).

## 1. Инициализация и Фильтрация
**Файлы:** `cmd/aitriage/scan.go` → `internal/core/context.go`

* Принимает путь. Парсит физический `.gitignore` в корне (через библиотеку `go-gitignore`).
* Обходит файловую систему, мгновенно отбрасывая бинарники и кэш (`node_modules`, `.git`), создавая чистый слайс `ctx.Files ([]*FileInfo)` для анализа.

## 2. Поднятие правил в память
**Файлы:** `internal/engine/default_rules.yaml` → `internal/config/config.go`

Парсит более 20 YAML-правил. Правила делятся на типы `target`:
* **ast**: глубокий поиск по абстрактному синтаксическому дереву.
* **code**: поиск по очищенной копии кода (без комментариев и логов).
* **docs**: поиск "остаточных галлюцинаций" строго внутри комментариев.
* **filename**: проверки на наличие конфигураций.

## 3. Big-O Экзекутор (Single-Pass Pipeline)
**Файл:** `internal/engine/engine.go` (Метод `Engine.Run`)

Выполняется строгий $O(N)$ цикл. Файл загружается в оперативной памяти (RAM) ровно один раз. На каждый файл натравливается параллельный пул горутин.

### Шаг 3.1: Tree-Sitter AST (Синтаксический анализ) 
**Файл:** `internal/core/fileinfo.go` → `GetTree()`
* Читает файл, если это JS/TS/Py/Go, вызывает `go-tree-sitter`.
* Строит AST (абстрактное синтаксическое дерево).
* Разделяет файл на три буфера в памяти: `RawCode`, `StrippedCode` (код без комментов и строковых литералов), `DocsOnly` (только комменты).
* **Как ищет:** Вместо текстового поиска `exec(`, он делает структурный AST-запрос: найти `call_expression`, где функция это `exec`. Не фолсит на строках вроде `console.log("exec")`.

### Шаг 3.2: Математический поиск секретов (Shannon Entropy) 
**Файл:** `internal/engine/engine.go` (Метод `evaluateEntropy`)
* Ищет в коде строки 16-128 символов (`[a-zA-Z0-9+/=_-]`).
* Вычисляет математическую энтропию (Shannon formula: суммирование `p * math.Log2(p)`).
* **Как ищет:** Если плотность случайных символов выше 4.5 бит (что означает сложную рандомизацию, свойственную `API_KEY` или `JWT`), триггерит критикал-уязвимость, даже если ИИ назвал переменную неприметно (`var a = '...'`).

### Шаг 3.3: Поиск Entropy-Галлюцинаций (Regex Layer)
**Файл:** `internal/engine/engine.go` (Метод `evaluateEntropy` / `evaluateFile`)
* Прогоняет регулярки по узкоспециализированным буферам памяти.
* **Как ищет:** Например, `ENTR-04` (извинения ИИ) ищет фразу `"As an AI"` строго в буфере `DocsOnly`. `ENTR-12` тупо проверяет, превышает ли файл 1500 строк ("God File" свалка). А по файлу `.cursorrules` отлавливаются попытки запретить валидацию ИИ.

### Шаг 3.4: Присутствие Архитектуры (Project Rules)
* Сканирует директорию на наличие требуемых расширений (`.sum`, `.lock`) и middleware компонентов по всему скоупу проекта. Отсутствие Lock-файла засчитывается как категорическая уязвимость (Dependency Hijacking).

## 4. Агрегация и Скорринг
**Файлы:** `internal/scorer/scorer.go` → `internal/reporter/reporter.go`

* Собирает массив `[]core.CheckResult`.
* Считает пенальти: Critical (-25), High (-15), Medium (-10).
* Конвертирует балльные вычеты в финальный `SecurityGrade` (от A+ до F).
* Мапит данные в международный JSON-стандарт `SARIF 2.1.0` и отдает как Report, который может переварить любой CI/CD пайплайн (GitHub Advanced Security, GitLab). Если запущен в консоли — рисует ASCII дерево и отдает `Exit code 1`. Утечек во внешнюю сеть нет.
