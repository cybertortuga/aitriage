# github-action-optimization

## Обзор
Текущая реализация GitHub Action (`action.yml`) использует директиву `image: 'Dockerfile'`. Это заставляет раннеры GitHub Actions пересобирать весь образ при каждом запуске: устанавливать Node.js, собирать Web UI, скачивать C-зависимости, компилировать Go с `CGO_ENABLED=1` и ставить внешние Python/Go сканеры (semgrep, bandit, gitleaks, trivy). Это занимает ~10-15 минут на каждый запуск пользовательского CI.
Согласно аудиту KII (`aitriage_status_report.md.resolved`), релизные бинарники из GoReleaser собираются без AST (`CGO_ENABLED=0`), поэтому мы не можем сделать легковесный Composite Action со скачиванием бинарника — он будет без ключевого функционала.
Решение: создать CI/CD пайплайн для сборки и публикации готового Docker-образа в GitHub Container Registry (GHCR) и перевести `action.yml` на использование этого pre-built образа. Время работы Action сократится до пары секунд (только `docker pull`).

## Задача 1: Создание Workflow для публикации GHCR
### Описание
Создать пайплайн GitHub Actions, который будет собирать `Dockerfile` и публиковать образ `ghcr.io/cybertortuga/aitriage` при пушах в `main` и создании тегов релизов.
### Затронутые файлы
- `.github/workflows/docker-publish.yml`
### Подзадачи
- [ ] Создать файл пайплайна с использованием `docker/build-push-action`.
- [ ] Настроить авторизацию в GHCR через `secrets.GITHUB_TOKEN`.
- [ ] Добавить генерацию тегов (`latest` для `main`, и семантические `v1`, `v1.x.x` для релизов).
- [ ] Включить кэширование Docker-слоев (`type=gha`), чтобы ускорить саму сборку.
### Критерии приёмки
- [ ] При пуше в `main` образ автоматически собирается и пушится в GHCR.
- [ ] При релизе собираются соответствующие семантические теги.
### Риски
- Сборка может быть медленной даже с кэшем, но это будет происходить только в репозитории AITriage, а не у конечных пользователей.

## Задача 2: Оптимизация action.yml
### Описание
Перевести экшен с локальной сборки на pre-built образ из GHCR.
### Затронутые файлы
- `action.yml`
### Подзадачи
- [ ] Заменить `image: 'Dockerfile'` на `image: 'docker://ghcr.io/cybertortuga/aitriage:latest'` (или мажорную версию, если таковая будет принята).
- [ ] Проверить, что параметры `args` (scan, format, target-stack) корректно передаются в ENTRYPOINT образа.
### Критерии приёмки
- [ ] `action.yml` больше не запускает сборку Dockerfile "на лету".
- [ ] Аргументы успешно пробрасываются в исполняемый бинарник `aitriage` внутри контейнера.
### Риски
- Завязка на `latest` тег может сломать интеграции пользователей при breaking changes. Необходимо будет зафиксировать инструкцию для пользователей использовать стабильные теги версий.

## Риски и зависимости
- **Видимость GHCR (Visibility)**: Пакет в GHCR по умолчанию публикуется как приватный. После первой успешной сборки и пуша образа, владельцу репозитория (Organization Owner) необходимо зайти в настройки пакета на GitHub и изменить его видимость на Public. Иначе пользователи (в своих репозиториях) будут получать ошибку `unauthorized` при попытке загрузить Docker Action.
- **GoReleaser CGO_ENABLED=0**: В KII-отчете указано, что бинарники собираются без CGO (без AST). Эта проблема остается для CLI-пользователей, скачивающих бинарники напрямую, однако Docker-контейнер (который будет использоваться Action) собирается из `Dockerfile`, где `CGO_ENABLED=1` уже настроен корректно. Action будет обладать полным функционалом.
