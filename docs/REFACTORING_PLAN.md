# AITriage: План Рефакторинга и Развития

Этот документ описывает переход проекта от MVP (с регулярными выражениями и O(N*M) обходами файлов) к Production-Ready инструменту. "Швейцарский нож", который в одном бинарнике совмещает проверку архитектурных security-практик (P-0X) и качества AI-кода (ENTR-XX).

## Фаза 1: Ядро — Единый `ProjectContext`
Цель: Убить O(N*M) сканирование, убрать утечки памяти и хардкод в 500 файлов.

- [x] **Задача 1.1:** Создать пакет `internal/core` и структуру `ProjectContext`.
- [x] **Задача 1.2:** Реализовать однопроходное сканирование (кеширование путей, содержимого, lazy-loading файлов).
- [x] **Задача 1.3:** Реализовать умный `IgnoreEngine` (чтение `.gitignore` и базовый фильтр `node_modules`, `.next`, `dist`, `venv`).
- [x] **Задача 1.4:** Обновить интерфейс `checker.Checker` — передавать `*core.ProjectContext` вместо `projectPath string`.
- [x] **Задача 1.5:** Рефакторинг всех существующих чекеров на использование `ProjectContext`.

## Фаза 2: ENTR-движок (Smart Regex и Предварительная обработка)
Цель: Резко снизить False Positives для ENTR-проверок без раздувания бинарника тяжелыми AST-библиотеками.

- [x] **Задача 2.1:** Реализовать `Stripper` (вырезание комментариев и строк из кода).
- [x] **Задача 2.2:** Улучшить `ProjectContext`, добавив методы `GetStrippedContent(filePath)` (только код) и `GetDocsOnlyContent(filePath)` (только комментарии и строки).
- [x] **Задача 2.3:** Перевести ENTR-чекеры на использование предварительно обработанного контента.

## Фаза 3: Тонкая настройка чекеров (Калибровка логики)
Цель: Сделать каждую проверку пуленепробиваемой.

- [x] **Задача 3.1:** `IdentityLeak` и `Placeholders` — заставить искать **только** в `GetDocsOnlyContent()`.
- [x] **Задача 3.2:** `JWTBlindspot` — искать в `GetStrippedContent()`. Улучшить логику проверки (verify вызван хоть где-то до / после).
- [x] **Задача 3.3:** `ClientSecretLeak` — обновить регулярное выражение для поддержки деструктуризации (например, `const { SECRET } = process.env`).
- [x] **Задача 3.4:** `TS Surrender` — считать процент `any` и `@ts-ignore` по отношению к размеру файла, а не просто "3 вхождения".

## Фаза 4: Умный детектор стека (Монорепозитории)
Цель: Поддержка сложных проектов (Next.js + FastAPI в одной репе).

- [x] **Задача 4.1:** Научить `ProjectContext` находить несколько фреймворков в одном проекте (например, если есть `package.json` в `frontend/` и `requirements.txt` в `backend/`).
- [x] **Задача 4.2:** Обновить `main.go`, чтобы он запускал ENTR-чекеры для **всех** обнаруженных фреймворков.
- [x] **Задача 4.3:** Добавить `framework` тег в `CheckResult` (для идентификации источника ошибки в монорепозиториях).
- [x] **Задача 4.4:** Обновить систему репортинга для вывода результатов по подпроектам.
