# ci-cd-audit

## Обзор
Цель задачи — провести аудит и интеграцию локальных изменений в удалённые репозитории для проверки работоспособности CI/CD пайплайна (AITriage Security Pipeline) на GitHub Actions. Мы проверяем, что дублирование таблиц устранено флагом `--no-summary`, генерируется ровно одна Markdown-таблица формата GFM с колонками "Triage Status" и "Rationale", а также пайплайн не падает из-за некорректных триггеров или отсутствующих API-ключей (если это не предусмотрено конфигурацией).

## Затронутые файлы и области
- Локальный репозиторий `cybertortuga/aitriage` (ожидают коммита: `cmd/aitriage/scan.go`, `internal/agent/prompts/templates.go`)
- Локальный репозиторий `cybertortuga/aitriage-test-vulnerabilities` (ожидает коммита: `.github/workflows/aitriage.yml`)
- Инфраструктура GitHub Actions (workflow runs).

## Задача 1: Интеграция изменений в aitriage
### Описание
Зафиксировать локальные изменения кода сканера и промптов, и отправить их в удалённый репозиторий `aitriage` в ветку `main`. Это спровоцирует запуск `docker-publish.yml` и обновление образа `ghcr.io/cybertortuga/aitriage:latest`, который используется в экшене.
### Затронутые файлы
- `cmd/aitriage/scan.go`
- `internal/agent/prompts/templates.go`
### Подзадачи
- [x] Выполнить `git commit` и `git push` в репозитории `aitriage`.
- [x] Проверить статус сборки Docker образа (через `gh` cli или браузер).
### Критерии приёмки
- [x] Коммит успешно доставлен в `main`.
- [x] Docker образ `latest` обновлен на GHCR.
### Риски
- Сборка образа может занять несколько минут, нужно обязательно дождаться её окончания перед переходом к следующей задаче.

## Задача 2: Обновление конфигурации тестового репозитория
### Описание
Зафиксировать добавление флага `--no-summary` в workflow-файл тестового репозитория и отправить изменения.
### Затронутые файлы
- `.github/workflows/aitriage.yml` (в `aitriage-test-vulnerabilities`)
### Подзадачи
- [x] Выполнить `git commit` и `git push` в `aitriage-test-vulnerabilities`.
### Критерии приёмки
- [x] Коммит успешно доставлен в `main`.
### Риски
- Конфликты версий, если кто-то другой параллельно менял workflow-файл.

## Задача 3: Запуск и аудит CI/CD пайплайна
### Описание
Запустить вручную пайплайн `AITriage Security Pipeline` в `aitriage-test-vulnerabilities` и проанализировать результат на соответствие требованиям.
### Затронутые файлы
- GitHub Actions Logs
### Подзадачи
- [x] Запустить workflow `aitriage.yml` вручную (`workflow_dispatch`).
- [x] Дождаться завершения workflow.
- [x] Получить и проанализировать Step Summary.
### Критерии приёмки
- [x] В логах и summary присутствует ровно одна красиво отформатированная таблица (GFM).
- [x] В таблице присутствуют колонки `Triage Status` и `Rationale`.
- [x] Дата в отчёте соответствует текущей.
### Риски
- Ошибка "Red cross" из-за недостатка API ключа (GEMINI_API_KEY) или другой конфигурационной ошибки.

## Риски и зависимости
Общая зависимость от доступности GitHub Container Registry. Тестовый репозиторий ссылается на экшен `@v1` (который внутри жестко завязан на тег `:latest` докера). Если Docker-образ `:latest` не успеет обновиться, тестовый пайплайн запустит старый код. Необходимо строго контролировать последовательность шагов.
