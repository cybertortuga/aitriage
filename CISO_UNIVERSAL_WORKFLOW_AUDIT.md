# Universal GitHub Actions Workflow — CISO Readiness Audit

## Purpose and change-control boundary

This is the working record for selecting, auditing, and hardening the primary
consumer workflow: `examples/github-actions/aitriage-security.yml`.

The workflow must be credible as a production security control: deterministic
scanning collects evidence without blocking on untriaged findings, then
mandatory AI triage is the sole merge authority. The AI policy applies only
after false positives have been removed from the Health Check.
No remote release, tag movement, or workflow installation into a consuming
repository is in scope.

### Scope extension — sole-operator GitHub Actions control

The repository owner requires that no person other than the designated operator
can cause GitHub Actions jobs to execute in this repository. This cannot be
guaranteed by a fork-PR guard alone: existing workflows, direct pushes, manual
dispatch permissions, branch protection, and repository access must be treated
as one control surface.

No GitHub username is hard-coded in workflow YAML. The designated operator must
be supplied by repository variable `AITRIAGE_ALLOWED_ACTOR_ID`; this avoids
silently granting authority to an assumed account name.

## Scope

- Primary workflow: `examples/github-actions/aitriage-security.yml`
- Published Docker Action contract: `action.yml` and `entrypoint.sh`
- Release image mapping: `.github/workflows/docker-publish.yml`
- Policy semantics: `cmd/aitriage/scan.go` and `internal/report/healthcheck/`
- AI trust boundary: `cmd/aitriage/agent.go` and its prompt construction

## Non-goals

- Do not alter the three specialised example workflows unless audit evidence
  proves they share a defect with the primary workflow.
- Do not use `pull_request_target` to make secrets available to forked PRs.
- Do not make raw scanner output capable of blocking before AI triage.
- Do not publish, retag, or deploy an Action release without separate approval.
- Do not claim that YAML alone can stop a repository administrator or a user
  allowed to change protected workflow files.

## Acceptance criteria

- [x] Every `uses:` dependency in the primary workflow has an auditable,
      immutable reference or a documented release-management exception.
- [x] The Action runtime image is not selected through `:latest`.
- [x] Static scanning is evidence-only and cannot block on raw findings.
- [x] SARIF upload is evidence-only and preserves a downloadable fallback.
- [x] The GitHub Job Summary is emitted only after completed AI triage; raw
      scanner output cannot appear there first.
- [x] Fork PRs cannot access the LLM secret or initiate an LLM-backed scan.
- [x] Fork PRs cannot execute the deterministic scanner or any other job.
- [x] AI triage is mandatory for every authorised run; an AI failure fails the
      pipeline and cannot be hidden by an advisory fallback.
- [x] The sole policy gate fails on any post-triage true positive or a
      post-triage Health Score below 70.
- [x] Reports have explicit retention and do not silently disappear.
- [x] YAML and GitHub Actions expression syntax pass `actionlint`.
- [x] A real SARIF file generated by the current CLI parses successfully.
- [x] Relevant Health Check and command-policy tests pass.

## Decision log

| Decision | Rationale | Status |
|---|---|---|
| Make `aitriage-security.yml` the universal example | It has the complete deterministic + mandatory-AI control flow; specialised workflows remain task-specific. | Accepted |
| No scheduled trigger | Every code change already invokes the workflow; periodic scanning is not justified unless the organisation explicitly needs feed-driven reassessment. | Accepted |
| Static collection and AI triage both run; AI is the sole policy gate | Raw scanner results can be false positives, so collection never blocks. The mandatory agent classifies them, then fails on any remaining true positive or Health Score below 70. | Accepted |
| AI is skipped for fork PRs | Avoid provider-secret exposure and uncontrolled code-data egress. | Accepted |
| All jobs are skipped for fork PRs | The user does not permit external PR code to run in this repository at all. A job-level condition prevents checkout and container execution while retaining same-repository PR checks. | Accepted |
| Use `github.actor_id`, not a username | GitHub account IDs are stable identifiers. The allowlist is externalised as a repository variable so source does not assume or expose the operator's account name. | Accepted |
| Fail closed when the allowlist is absent | An unset `AITRIAGE_ALLOWED_ACTOR_ID` makes every guarded job skip. This is safer than silently granting execution to all write collaborators. | Accepted |

## Release prerequisite and operating checklist

The local source is ready, but no consumer receives the Docker digest change
until it is released. This is intentionally outside the scope of this audit.

1. Review and commit `action.yml`, the primary example, and this audit record.
2. Build and publish the reviewed Action image, then create a release whose
   Action metadata contains the pinned image digest.
3. Advance the public `v1` reference only to that reviewed release commit; do
   not move it independently of the matching image/release evidence.
4. In a target repository, copy the primary example to
   `.github/workflows/aitriage.yml` and run `workflow_dispatch` once.
5. Configure branch protection to require **AI Triage & Fix Specs**. Do not
   make deterministic collection a required check: it is evidence-only.
6. Enable GitHub Code Security if SARIF in Code Scanning is desired. SARIF
   upload remains reporting-only; the downloadable artifact is retained either
   way.
7. Create environment `ai-triage`; store `GEMINI_API_KEY` there, restrict
   eligible branches, and require reviewer approval where the GitHub plan
   supports it. Treat LLM triage as approved source-data egress.
8. Confirm the 14-day artifact retention satisfies the organisation's evidence
   policy; lower it if reports contain material that should be retained less.

## Sole-operator setup checklist

This is required before relying on the new guards:

1. Obtain the numeric GitHub account ID for the sole operator, for example with
   `gh api user --jq .id` while authenticated as that account.
2. In **Settings → Secrets and variables → Actions → Variables**, create
   `AITRIAGE_ALLOWED_ACTOR_ID` with that numeric value. It is not a secret.
3. Keep the workflow triggers if the operator's own pushes/PRs should start
   checks automatically. An event created by anyone else will create, at most,
   a workflow record whose jobs are all skipped before a runner is assigned.
4. If even a skipped workflow record is unacceptable, remove `push` and
   `pull_request` triggers and retain only `workflow_dispatch`. This still
   requires GitHub access control, because every writer can click Run workflow.
5. Remove every other collaborator's write/admin access. Protect `main` and
   require review for `.github/workflows/**`, `action.yml`, and release files;
   otherwise a collaborator who can change those files can remove the guard.
6. In GitHub Actions settings, keep the default `GITHUB_TOKEN` read-only and
   allow only required actions. Job-level YAML guards do not constrain an
   administrator who can change repository settings or workflow source.

## Final audit conclusion

`examples/github-actions/aitriage-security.yml` is the designated universal
workflow. Deterministic scanning collects non-blocking evidence; mandatory AI
triage is the sole policy gate and fails on every surviving true positive or a
post-triage Health Score below 70. The Job Summary is written only after AI
triage completes, even when the resulting policy gate fails, and consists of the AI-generated human assessment,
remediation prompt, and AI Agent Data. SARIF remains an evidence/integration path.
The public `@v1` Action still uses its prior metadata/image until a reviewed
release is published. That release must include WF-001, WF-010, and WF-011
before the current workflow can be represented as the public CISO-ready build.
WF-013 must be resolved or isolated first, because the production image cannot
currently be built.
